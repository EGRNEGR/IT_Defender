## 1) Product overview (what we are building)

### 1.1 Purpose
- The extension locally analyzes user-entered text **inside web pages** to detect potentially sensitive information (PII/secrets).
- When sensitive data is detected, it:
  - Visually highlights the risky input field (outline and subtle glow).
  - Shows a small ÔÇťshieldÔÇŁ floating UI near the field with a count of findings.
  - Optionally **blocks message sending** (strict mode) or **auto-masks** sensitive values before sending.

### 1.2 Where it operates (scope modes)
The extension supports 3 scope modes (configured via Options UI and reflected in Popup UI):
- **LLM-only** (default): runs only on known LLM chat domains:
  - `chatgpt.com`
  - `chat.openai.com`
  - `claude.ai`
  - `gemini.google.com`
- **Global**: runs on all websites.
- **Custom allowlist**: runs only on user-provided hostnames/domains.

### 1.3 Exceptions (blocklist)
Even in Global mode, the extension can be disabled on a list of ÔÇťexceptionÔÇŁ sites (defaults aimed at email/login/docs/forms):
- Enabled by default.
- Default exception hosts include:
  - `mail.google.com`, `outlook.live.com`, `mail.yahoo.com`, `proton.me`
  - `accounts.google.com`, `login.live.com`, `id.apple.com`
  - `docs.google.com`, `forms.gle`

The exception logic must match exact host or subdomain match:
- `example.com` matches `example.com` and `sub.example.com`.

## 2) Functional requirements (detailed)

### 2.1 Supported input fields to analyze
Monitor these editable surfaces:
- `textarea`
- `input`
- `[contenteditable="true"]` (LLM chat editors often use this)

Define a single selector constant:
- `FIELD_SELECTOR = '[contenteditable="true"], textarea, input'`

### 2.2 Events to analyze
- On user typing and paste:
  - Use capturing listeners (`true`) for robustness.
  - For each `input`/`paste` event, find the closest element matching `FIELD_SELECTOR` and analyze its current text.

### 2.3 Dynamic DOM support
Modern LLM sites insert fields dynamically. Use a `MutationObserver` to:
- Watch `document.documentElement` with `{ childList: true, subtree: true }`.
- For any added node:
  - If it matches `FIELD_SELECTOR`, attach listeners.
  - Also scan inside it for nested fields and attach listeners.

### 2.4 Detection types
Implement a local detector module (no dependencies) that finds:
- **Email addresses**
- **Common API keys / tokens** by known prefixes:
  - OpenAI style `sk-...`
  - GitHub `ghp_...`
  - Slack bot `xoxb-...`
  - Google API key `AIza...`
- **JWT tokens** in `header.payload.signature` form (base64url-ish, common `eyJ` prefix)
- **Credit card numbers** (13ÔÇô19 digits, allowing spaces/dashes) validated using the **Luhn algorithm**

Additionally, include patterns for future expansion (even if disabled by default in settings UI):
- Phone numbers
- AWS access keys (`AKIA...`)
- Private key blocks (`-----BEGIN ... PRIVATE KEY----- ... -----END ... PRIVATE KEY-----`)

### 2.5 Findings model
The detector returns an ordered array of findings:
- Each finding includes:
  - `type` (e.g. `email`, `apiKey`, `jwt`, `card`, `phone`, `awsAccessKey`, `privateKey`)
  - `match` (raw matched text)
  - `start` / `end` positions (character offsets)
  - For cards: `normalized` (digits only, no separators)
- Sort findings ascending by `start`.

### 2.6 Masking
Provide a `maskText(text, findings)` function that returns the same text but with findings replaced by placeholders:
- `[HIDDEN_EMAIL]`
- `[HIDDEN_API_KEY]`
- `[HIDDEN_JWT]`
- `[HIDDEN_CARD]`
- `[HIDDEN_PHONE]`
- `[HIDDEN_AWS_KEY]`
- `[HIDDEN_PRIVATE_KEY]`
- Unknown types fallback to `[HIDDEN]`

Masking must be stable:
- Iterate through findings left-to-right.
- Skip overlapping findings (if `finding.start < cursor`).

### 2.7 Field state tracking
Maintain per-field state with a `WeakMap<Element, State>` storing:
- `findings`
- `originalText`
- `maskedText`

Whenever analysis runs:
- Compute findings with detector.
- Filter findings according to enabled scan toggles (email/apiKey/jwt/card).
- Update `STATE` and render the UI state for that field.

### 2.8 Visual highlight for risky fields
When field has findings:
- Set `element.style.outline = "2px solid #b91c1c"`
- `outlineOffset = "2px"`
- Set attribute `data-ai-privacy-flagged="true"`
- Also include a CSS rule to add subtle red glow:
  - `[data-ai-privacy-flagged="true"] { box-shadow: 0 0 0 2px rgba(185, 28, 28, 0.2); transition: box-shadow 120ms ease; }`

When field is clean:
- Remove outline styles and attribute.
- Hide shield UI.

### 2.9 Floating shield UI (always isolated)
Create a UI manager that injects UI into a `ShadowRoot` mounted on `document.documentElement`.
Requirements:
- Root host element id: `ai-privacy-sentinel-root`
- Use `attachShadow({ mode: "open" })`
- Inject all UI styles inside the shadow root to avoid site CSS interference.

Shield button:
- `position: fixed; z-index: 2147483647; pointer-events: auto`
- Red background `#8b0000`, white text.
- Displays: `­čŤí đŁđ░đ╣đ┤đÁđŻđż: <count>`
- Positioned near the active field rect:
  - `top = max(anchorRect.top + scrollY - 44, 8)`
  - `left = max(anchorRect.right + scrollX - 150, 8)`

### 2.10 Modal behavior (strict/auto-mask-confirm)
A modal backdrop + card must also live inside the shadow root (same top z-index).
Modal requirements:
- Backdrop: semi-transparent dark overlay.
- Modal card: white background, dark text, rounded corners.
- Buttons:
  - Primary ÔÇťmask & sendÔÇŁ (red background).
  - Secondary ÔÇťcancelÔÇŁ (neutral background).

Modal must support 2 use-cases:
1) **Strict mode** (default):
   - When user tries to send and findings exist, block sending and show modal:
     - Title: ÔÇťPotentially sensitive data detectedÔÇŁ
     - Body: a summary string such as ÔÇťFound: email addresses: 1, API keys: 2. Mask data before sending.ÔÇŁ
     - Primary: ÔÇťMask and sendÔÇŁ
     - Secondary: ÔÇťCancelÔÇŁ
2) **Auto-mask with confirmation**:
   - If auto-mask is enabled and confirm is enabled:
     - Block send and show modal asking:
       - ÔÇťSensitive data found. Replace it with masks before sending?ÔÇŁ
     - On confirm: mask field, then re-trigger sending.

### 2.11 Intercepting ÔÇťSendÔÇŁ
You must intercept both:
- Click on send buttons:
  - `SEND_SELECTOR = '[aria-label="Send"], button[type="submit"]'`
  - Use `document.addEventListener("click", interceptSend, true)`
  - In handler: `event.target.closest(SEND_SELECTOR)`
- Enter key for chat inputs:
  - `document.addEventListener("keydown", handleKeydown, true)`
  - Only treat as send when:
    - `event.key === "Enter"`
    - `!event.shiftKey`
    - target is a field in `FIELD_SELECTOR`

To identify which field to analyze for a send button:
- If button is inside a `<form>`, choose the first field within the form that has non-empty text, else fallback to first field.
- If no form, search all fields similarly.

Blocking rules:
- If findings exist and:
  - `autoMask === true`:
    - If `autoMaskConfirm === true`: block, show confirm modal, then mask and proceed.
    - Else: mask immediately and allow sending (do not block further).
  - Else if `strict === true`: block and show strict modal.
  - Else (soft mode): do not block; only highlight + shield.

When blocking:
- `event.preventDefault()`
- `event.stopImmediatePropagation()`

When proceeding after masking:
- Trigger the send button again via `button.click()` (for click flow).
- For enter-key flow, after masking you can keep focus in the field (or attempt to trigger send depending on site; minimal behavior is acceptable as long as it doesnÔÇÖt double-send).

### 2.12 Settings storage
Store settings in `chrome.storage.sync` under key `"aiPrivacySettings"`.
Default settings shape:

```js
{
  scan: { email: true, apiKey: true, jwt: true, card: true },
  mode: { strict: true, autoMask: false, autoMaskConfirm: false },
  scope: { mode: "llm", customHosts: [] }, // "llm" | "global" | "custom"
  exceptions: { enabled: true, hosts: [/* default list */] }
}
```

Merging rules:
- When loading stored settings, merge into defaults and validate types.
- Unknown keys are ignored.

### 2.13 Runtime settings application
Content script must:
- Load settings at startup (if storage available).
- Apply settings and only start listeners if the current page is ÔÇťin scopeÔÇŁ.

In-scope logic:
- If exceptions enabled and current host matches exception list => not in scope.
- Else:
  - `scope.mode === "global"` => in scope
  - `scope.mode === "custom"` => in scope only if host matches customHosts (exact or subdomain)
  - Otherwise (`"llm"`) => in scope only if host is in LLM host set

### 2.14 Popup behavior (quick status)
The extension must have a popup showing:
- Brand name + shield icon
- Current hostname
- A pill saying ON/OFF based on the current hostname and settings
- A short explanation: e.g. ÔÇťMode: LLM chats onlyÔÇŁ or ÔÇťException: analysis disabled for this site.ÔÇŁ

Popup implementation notes:
- Use `chrome.tabs.query({ active: true, currentWindow: true })`
- Read settings via `chrome.storage.sync.get("aiPrivacySettings")`
- Use `chrome.scripting.executeScript` to evaluate a function in the tab that returns `window.location.hostname`.

### 2.15 Options page (full settings UI)
Options page must include:
- **Scope** radio group (Global / LLM-only / Custom) and a textarea for custom hosts.
- **Data types** toggles for email/apiKey/jwt/card.
- **Exceptions** toggle + textarea list.
- **Pre-send behavior** toggles:
  - strict mode
  - auto mask
  - confirm before auto mask
- Any change should auto-save to storage and show a small ÔÇťSettings savedÔÇŁ status message.
